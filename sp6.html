Debug Tool: Line Drawer

In our previous projects made with Tga2D we had a convenient debugging tool to draw lines between 2 points or draw shapes such as circles and squares. I wanted to recreate a similar feature for IronWrought. To do so I created a system with a structure and interface similar to how IronWrought handles model creation.
CLine that holds DirectX11 data for topology, vertices, input layout and more.
CLineInstance that holds a pointer to a CLine along with a transform.
CLineFactory which creates a CLine based on a pre-determined shape or a set of points that describe a polygon.
A forward renderer that handles only CLineInstances.

I created functions for the shapes of the colliders we used for the project: square, circle and triangle. When the navigation mesh was being worked on I was requested to add support for visualizing it, resulting in the CreatePolygon() function.

There is room for improvement: streamlining creation of CLines and CLineInstances, support for more shapes and memory optimizations.

Modelviewer

A tool that I developed along with Emanuel Niklasson, from another team, for both our teams to use. A requirement for the project as well as a needed tool for the graphics artist to check how models looked in the engine. The tool was put together using existing systems with the goal being to get it up and running as fast as possible.

During development of SPITE: Bloodloss, our engine IronWrought was being put together alongside development of the game. At the time we didn't have a GUI to use with the engine, we had heard of ImGui but were unsure of the time it would take to implement it and how much use it was going to see. So we opted for using a console application as input for the tool.

To get all the FBX files from our asset directory we used std::recursive_directory_iterator, if a file with the correct extension was found it was added to a std::vector to hold all paths. Loading and swapping the model based on the index in the vector.

Later on a filter for skinned models was added so animations could be tested without having to be implemented in the game. Nicolas Risberg who was working on the VFX system had the idea to also support VFX playback through the modelviewer, he also suggsted adding a display option for the in-game HUD that auto played any animations it had.

Game Object and Component System

For IronWrought the team knew we wanted some form component system, some of us had created and worked with one in a previous project and were eager to work with it again.
A CGameObject holds any number of CComponents, each holding some logic. This was the core of the system we used, more functionalty was added later on.

We used Unity as a reference for the type of interface we wanted, with e.g: GetComponent(). GetComponent(), TryGetComponent(), RemoveComponent() all use typeid: const std::type_info& type = typeid(T);, to identify which component is requested.
Since a CComponent could require any number of parameters in its constructor so a variadic template was used to remove the need for separate functions to set data.
